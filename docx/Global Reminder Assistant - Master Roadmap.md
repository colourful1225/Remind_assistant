# Global Reminder Assistant - Master Roadmap

## 0. 项目一句话定义

这是一个 Android 应用，目标是让用户从任意文本信息中，快速创建：

1. **提醒 / 待办**
2. **系统日历事件**

并通过 **标准分享入口 + 文本处理入口 + 剪贴板建议 + 辅助功能增强入口**，逐步接近你想要的“全局快速加入提醒/日历”的体验。

---

# 1. 项目总目标

## 1.1 最终体验目标

用户在任何 app 中看到一段信息，例如：

* “明天下午5点去北区取快递”
* “周三下午3点在工大北区开组会”
* “下周一前提交实验报告”

都可以尽量低成本地完成以下动作之一：

* 加入备忘录并设置提醒
* 加入系统日历并快速确认
* 在微信/短信等场景中通过快捷入口进入上述流程

---

## 1.2 产品核心价值

本项目真正解决的问题不是“做一个提醒 app”，而是：

**把散落在聊天、短信、网页、通知中的信息，快速转化为可执行提醒。**

---

# 2. 产品能力总图

项目整体能力可以理解为五层：

## 第 1 层：基础容器

* 本地提醒
* 本地通知
* 系统日历接入

## 第 2 层：信息导入

* 分享文本
* 选中文本处理
* 手动输入

## 第 3 层：信息理解

* 文本清洗
* 时间识别
* 地点识别
* 标题生成
* 类型判断（提醒 / 日历）

## 第 4 层：轻量主动入口

* 剪贴板建议
* 去重与冷却

## 第 5 层：增强场景入口

* 辅助功能 PoC
* 微信 / 短信长按候选识别
* 快捷入口

---

# 3. 总体开发原则

## 3.1 先做标准能力，再做增强能力

顺序必须是：

1. 工程骨架
2. 导入链路
3. 提醒闭环
4. 日历闭环
5. 剪贴板建议
6. 辅助功能 PoC

不要反过来。
因为辅助功能是最不稳定、最重适配、最依赖已有基础链路的部分。

---

## 3.2 一切入口都复用统一导入流程

无论来源是：

* 分享
* 选中文本
* 剪贴板
* 辅助功能

都尽量转成：

`ImportRequest -> Parse -> ImportConfirm -> ReminderEditor / CalendarEditor`

不要做四套平行流程。

---

## 3.3 先闭环，再优化

优先让能力真正可用：

* 能导入
* 能保存
* 能提醒
* 能进系统日历

之后再优化：

* 识别率
* 体验
* UI
* 兼容性

---

## 3.4 平台边界必须诚实

尤其要始终记住：

* 普通 Android app 不能保证给所有第三方 app 原生长按菜单直接插入一个菜单项
* 微信等 app 只能通过“辅助功能等价入口”逼近体验
* 系统日历 Intent 比 Provider 直写更适合 MVP
* 厂商 ROM 差异会影响辅助功能和后台提醒稳定性

# 4. 总路线图

---

## Milestone 1：工程基础骨架

### 目标

建立可运行、可扩展的 Android 工程基础架构。

### 产出

* Kotlin + Compose + MVVM + Hilt + Room
* 导航骨架
* 5 个核心页面骨架
* 核心 domain model
* Reminder 数据表
* 首页读取本地提醒

### 为什么必须先做

这是所有后续功能的地基。没有这一层，后面所有能力都会越做越乱。

### 完成后你应该得到

* 一个真正能跑的 app
* 清晰的目录结构
* 可以继续往里塞导入、解析、提醒调度能力

### 风险

* AI 容易偷懒写成 demo 结构
* 容易把 ViewModel / Repository / DAO 混在一起

### 验收重点

* 是否真能运行
* 是否真接了 Room
* 首页是否真从数据库读
* ReminderEditor 是否真能保存基础数据

---

## Milestone 2：文本导入与基础解析

### 目标

打通“外部文本进入 app”的主链路。

### 产出

* ACTION_SEND 接入
* ACTION_PROCESS_TEXT 入口骨架
* ImportRequest 结构
* 基础 parser
* ImportConfirm 展示真实解析结果
* Reminder / Calendar 页面预填

### 为什么是第二步

没有导入能力，项目就只是一个普通提醒 app，不是你要的“全局信息转提醒”工具。

### 完成后你应该得到

* 从别的 app 分享文本到本 app
* 能看到原文与解析结果
* 能把解析结果送到 ReminderEditor 或 CalendarEditor

### 风险

* AI 可能把 intent 解析写死在 Activity 里
* 可能用不合理方式传递长文本参数
* parser 容易散落在 UI 层

### 验收重点

* 是否真能从其他 app 分享文本进入
* ImportConfirm 是否显示真实数据
* 预填是否真的生效
* parser 是否独立封装

---

## Milestone 3：提醒创建与本地通知闭环

### 目标

让提醒不只是“保存”，而是“真的会到时间提醒”。

### 产出

* Reminder 时间设置
* 保存 Reminder 到 Room
* AlarmManager + BroadcastReceiver 调度
* NotificationChannel + 本地通知
* 首页状态展示
* 删除 / 完成提醒
* 开机恢复基础版

### 为什么是第三步

到这一步，产品第一次真正“有用”。
你不再只是能导入文本，而是能把导入结果转成真正的提醒。

### 完成后你应该得到

* 创建提醒
* 设提醒时间
* 到点收到通知
* 首页能管理 reminder 状态

### 风险

* AI 容易用前台 timer 假装提醒
* 删除 / 完成后可能没取消调度
* NotificationReceiver 可能不回查数据库，导致已删除项仍提醒

### 验收重点

* 是否真靠系统调度
* app 不在前台时是否仍能提醒
* 删除/完成后是否不再提醒
* 重启后是否可恢复

---

## Milestone 4：日历编辑与系统日历集成

### 目标

打通“文本 -> 日历草稿 -> 系统日历新增页”的链路。

### 产出

* CalendarEditor 真正可编辑
* CalendarDraft 构建
* CalendarIntentFactory / Launcher
* 打开系统日历新增事件页
* 自动带入标题、时间、地点、描述
* 无日历 app 的降级处理

### 为什么在提醒闭环后做

因为提醒是你自己的闭环，而日历是系统集成。
先把自己的闭环跑通，再做系统集成，更稳。

### 完成后你应该得到

* 文本可以快速导向系统日历
* 用户只需确认，不用重新手填大部分信息

### 风险

* AI 可能误用 Calendar Provider 直写
* 可能乱要 READ/WRITE_CALENDAR 权限
* 可能没处理无日历 app 场景

### 验收重点

* 是否真打开系统日历新增页
* 字段是否真带入
* 是否没强依赖不必要权限
* 降级是否安全

---

## Milestone 5：剪贴板建议与低打扰主动入口

### 目标

做出一个现实、低摩擦、跨 app 覆盖面很广的轻量入口。

### 产出

* ClipboardMonitor
* 建议判断逻辑
* Suggestion history
* 去重与冷却
* 设置页相关开关
* 建议卡片 UI
* 复用既有导入链路

### 为什么这一步很关键

这是你第一个真正“接近全局入口”的能力。
虽然它不是改写所有长按菜单，但它能覆盖很多真实场景。

### 完成后你应该得到

* 复制一段高价值文本后，系统会建议加入提醒 / 日历
* 建议不会频繁骚扰
* 用户点击后直接进入既有导入流程

### 风险

* AI 容易做成“复制任何文本都弹”
* 去重 / 冷却只做内存版，杀进程后失效
* 设置开关只是 UI，不真正生效

### 验收重点

* 是否真的有过滤策略
* 是否真的做了持久化历史
* 设置是否真生效
* UI 是否轻量不打扰

---

## Milestone 6：辅助功能增强入口 PoC

### 目标

验证“微信 / 短信长按候选场景 -> 快捷入口 -> 导入链路”是否成立。

### 产出

* AccessibilityService
* 目标 app 匹配
* 候选场景识别
* 节点文本提取
* 快捷入口宿主
* SourceType.ACCESSIBILITY
* 设置页辅助功能说明与状态展示
* 冷却 / 去重

### 为什么它放在最后

因为这是最复杂、最脆弱、最重适配的一层。
没有前 1~5 步，它就算识别到了文本，也无处可去。

### 完成后你应该得到

* 至少在微信或短信某些场景中
* 当用户长按消息时，出现一个可点击的快捷入口
* 点击后把文本送入你已有的提醒 / 日历流程

### 风险

* 微信节点结构会变
* event 噪声巨大
* 误触发风险高
* ROM 对辅助功能与拉起 Activity 限制不一

### 验收重点

* 服务是否真能启用
* 是否不是把所有逻辑塞一个大文件
* 候选识别是否保守
* 文本提取是否合理
* 快捷入口是否能走既有导入链路

# 5. 各 Milestone 依赖关系

## 5.1 强依赖链

依赖顺序如下：

**M1 -> M2 -> M3 -> M4 -> M5 -> M6**

### 原因

* M1 提供工程骨架
* M2 提供统一导入链路
* M3 提供提醒闭环
* M4 提供日历闭环
* M5 复用导入链路做主动入口
* M6 复用导入链路做增强入口

---

## 5.2 哪些可以并行

理论上可以少量并行，但只建议在基础很稳之后：

### 可小规模并行

* M3 和 M4 的 UI 细节可以局部交叉
* M5 的 settings 存储可以和 M4 后半段少量并行
* M6 的教育页 / 设置页部分可以与 M5 后期并行

### 不建议并行

* M1 和 M2
* M2 和 M6
* M3 未完成就大做 M5/M6

# 8. 风险总表

## 8.1 产品风险

### 风险 A：把目标定义成“所有长按菜单都要原生插入提醒”

问题：

* 这在普通 Android app 层不现实
* 会让整个项目从一开始就卡死

正确做法：

* 定义为“提供尽可能统一、快速、低摩擦的提醒入口”

---

### 风险 B：过早上辅助功能

问题：

* 复杂
* 不稳定
* 难调试
* 还依赖已有链路

正确做法：

* 先把 M1-M5 打稳

---

## 8.2 技术风险

### 风险 C：AI 生成的工程结构不稳

表现：

* DAO 直连 UI
* ViewModel 手动 new Repository
* Activity 里处理全部 intent / parser / nav

规避：

* 每轮都强约束输出格式
* 先验收结构，再验收功能

---

### 风险 D：提醒调度假实现

表现：

* 用 delay / timer 假装提醒
* 只有 app 在前台时才提醒

规避：

* 明确要求 AlarmManager + Receiver
* 测试 app 退后台、杀进程、重启

---

### 风险 E：辅助功能误触发过多

表现：

* 在无关页面也弹快捷入口
* 噪声事件太多，体验很差

规避：

* 目标 app 白名单
* 候选场景规则
* 去重 / 冷却 / dismiss 强冷却

---

# 9. AI 开发管理建议

## 9.1 每次只给 AI 一个 Milestone 或一个子任务

不要一口气让 AI 实现 1 到 6。
最好的方法是：

* 先给 Milestone 总文档
* 再给当前 Milestone 任务包
* 一次只做一个阶段

---

## 9.2 每次要求 AI 按固定格式输出

固定要求它输出：

1. 任务理解
2. 架构设计
3. 文件树
4. 完整代码
5. Gradle 更新
6. Manifest 更新
7. 运行说明
8. 当前限制
9. 下一步建议

这样最稳。

---

## 9.3 先验收“结构”，再验收“功能”

拿到代码后，先看：

* 分层对不对
* 依赖注入对不对
* route / state / usecase 是否清晰

再看功能。

---

## 9.4 每个阶段都要做“真机/模拟器真实验证”

尤其这些阶段必须真测：

* M2：分享文本
* M3：定时通知
* M4：系统日历拉起
* M5：剪贴板建议
* M6：辅助功能事件与候选识别
